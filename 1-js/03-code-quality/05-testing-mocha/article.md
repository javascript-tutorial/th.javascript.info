# การทดสอบอัตโนมัติด้วย Mocha

ในงานต่อๆ ไปเราจะใช้การทดสอบอัตโนมัติ ซึ่งเป็นสิ่งที่ใช้กันแพร่หลายในโปรเจ็กต์จริงด้วย

## ทำไมต้องมีการทดสอบ?

เวลาเขียนฟังก์ชัน เรามักจินตนาการได้อยู่แล้วว่าควรจะทำงานอย่างไร — พารามิเตอร์แบบไหนให้ผลลัพธ์แบบไหน

ระหว่างพัฒนา เราตรวจสอบฟังก์ชันได้โดยการรันและเปรียบเทียบผลลัพธ์กับสิ่งที่คาดหวังไว้ เช่น อาจทำในคอนโซล

ถ้ามีอะไรผิดพลาด ก็แก้โค้ด รันใหม่ เช็คผลลัพธ์ วนไปแบบนี้จนกว่าจะใช้ได้

แต่การ "รันซ้ำ" ด้วยมือแบบนั้นยังไม่สมบูรณ์

**เมื่อทดสอบโค้ดโดยการรันซ้ำเอง มักจะมีบางอย่างหลุดไปได้ง่าย**

เช่น สมมติเรากำลังเขียนฟังก์ชัน `f` เขียนโค้ดนิดหน่อย ลองทดสอบดู — `f(1)` ใช้ได้ แต่ `f(2)` ไม่ทำงาน เราแก้โค้ดแล้วตอนนี้ `f(2)` ก็ใช้ได้แล้ว ดูเหมือนจะครบแล้วใช่ไหม? แต่เราลืมย้อนกลับไปเทสต์ `f(1)` ใหม่ ซึ่งอาจมี error โผล่ขึ้นมาก็ได้

นี่เป็นเรื่องปกติมาก ตอนพัฒนาอะไรสักอย่าง มักมีหลายกรณีที่ต้องคำนึงถึง แต่จะให้โปรแกรมเมอร์มานั่งเช็คทุกกรณีด้วยตัวเองทุกครั้งที่มีการเปลี่ยนแปลงนั้นเป็นเรื่องยาก เลยกลายเป็นว่าแก้อย่างหนึ่ง แล้วไปทำให้อีกอย่างพังได้ง่ายๆ

**การทดสอบอัตโนมัติหมายถึงการเขียนเทสต์แยกออกมาต่างหาก เป็นส่วนเสริมของโค้ด เทสต์เหล่านั้นจะรันฟังก์ชันของเราในหลากหลายวิธี และเปรียบเทียบผลลัพธ์กับสิ่งที่คาดหวัง**

## Behavior Driven Development (BDD)

มาเริ่มกันด้วยเทคนิคที่เรียกว่า [Behavior Driven Development](http://en.wikipedia.org/wiki/Behavior-driven_development) หรือเรียกสั้นๆ ว่า BDD

**BDD คือ 3 สิ่งในหนึ่งเดียว ทั้งเทสต์ เอกสาร และตัวอย่างการใช้งาน**

เพื่อทำความเข้าใจ BDD ลองมาดูตัวอย่างกรณีศึกษาจริงๆ กัน

## การพัฒนา "pow": ข้อกำหนด

สมมติว่าเราอยากเขียนฟังก์ชัน `pow(x, n)` ที่คืนค่า `x` ยกกำลัง `n` โดยที่ `n` เป็นจำนวนเต็มที่ `n≥0`

จริงๆ แล้วนี่เป็นแค่ตัวอย่าง เพราะ JavaScript มีตัวดำเนินการ `**` ที่ทำแบบนี้ได้อยู่แล้ว แต่ตรงนี้เราจะเน้นที่กระบวนการพัฒนา ซึ่งนำไปใช้กับงานที่ซับซ้อนกว่านี้ได้ด้วย

ก่อนจะเขียนโค้ดของ `pow` เราจินตนาการก่อนได้ว่าฟังก์ชันควรทำอะไร แล้วเขียนบรรยายไว้

คำบรรยายแบบนี้เรียกว่า *ข้อกำหนด (specification)* หรือเรียกสั้นๆ ว่า spec — ข้างในจะมีรายละเอียดกรณีการใช้งานต่างๆ พร้อมเทสต์สำหรับแต่ละกรณี ดังนี้

```js
describe("pow", function() {

  it("ยกกำลัง n", function() {
    assert.equal(pow(2, 3), 8);
  });

});
```

spec มีองค์ประกอบหลัก 3 ส่วนด้วยกัน

`describe("ชื่อหัวข้อ", function() { ... })`
: บอกว่าเรากำลังอธิบายฟังก์ชันอะไร ในกรณีนี้คือ `pow` — ใช้จัดกลุ่มบล็อก `it` ที่เป็น "ตัวทำงาน"

`it("คำอธิบายกรณีการใช้งาน", function() { ... })`  
: ในชื่อของ `it` จะบรรยาย *ในรูปแบบที่คนอ่านแล้วเข้าใจ* ถึงกรณีการใช้งานเฉพาะ และอาร์กิวเมนต์ตัวที่สองก็จะเป็นฟังก์ชันที่ทดสอบกรณีดังกล่าว

`assert.equal(value1, value2)`
: โค้ดภายในบล็อก `it` ถ้าอิมพลีเมนต์ถูกต้อง ควรรันได้โดยไม่เกิด error

    ฟังก์ชัน `assert.*` ใช้เช็คว่า `pow` ทำงานได้ตามที่คาดหวังหรือไม่ ตรงนี้เราใช้ `assert.equal` ซึ่งจะเปรียบเทียบอาร์กิวเมนต์สองตัว แล้วโยน error ถ้าไม่เท่ากัน — ในที่นี้คือเช็คว่า `pow(2, 3)` ได้ `8` รึเปล่า นอกจากนี้ยังมีฟังก์ชันตรวจสอบแบบอื่นๆ อีก ซึ่งเราจะเพิ่มภายหลัง

spec นี้รันได้เลย โดยจะไปรันเทสต์ที่อยู่ในบล็อก `it` — เดี๋ยวจะมาดูกันทีหลัง

## ขั้นตอนการพัฒนา

ขั้นตอนการพัฒนามักจะเป็นแบบนี้:

1. เขียน spec เบื้องต้น พร้อมเทสต์สำหรับฟีเจอร์พื้นฐานที่สุด
2. เขียนอิมพลีเมนต์เบื้องต้น
3. รันเทสต์ผ่านเฟรมเวิร์ก [Mocha](https://mochajs.org/) (จะอธิบายเพิ่มเร็วๆ นี้) เพื่อเช็คว่าใช้ได้ไหม — ถ้ายังไม่ครบถ้วนก็จะเห็น error แล้วค่อยทยอยแก้จนทุกอย่างผ่าน
4. ตอนนี้เรามีอิมพลีเมนต์เบื้องต้นที่ใช้ได้จริงพร้อมเทสต์แล้ว
5. เพิ่มกรณีการใช้งานเข้าไปใน spec — ซึ่งอิมพลีเมนต์อาจยังไม่รองรับ เทสต์จึงเริ่มล้มเหลว
6. กลับไปทำข้อ 3 — ปรับแก้อิมพลีเมนต์จนเทสต์ผ่านหมด
7. ทำข้อ 3-6 ซ้ำไปเรื่อยๆ จนทุกฟีเจอร์เสร็จสมบูรณ์

การพัฒนาจึงเป็นแบบ *วนซ้ำ (iterative)* — เขียน spec, อิมพลีเมนต์, เทสต์ให้ผ่าน แล้วก็เพิ่มเทสต์ วนไปเรื่อยๆ สุดท้ายเราจะได้ทั้งอิมพลีเมนต์ที่ใช้งานจริงได้ พร้อมกับเทสต์ครบถ้วน

ทีนี้มาดูกระบวนการนี้ในกรณีตัวอย่างของเรากัน

ขั้นตอนแรกเสร็จแล้ว — เรามี spec เบื้องต้นของ `pow` แล้ว ก่อนจะลงมือเขียนอิมพลีเมนต์ ลองใช้ไลบรารี JavaScript เพื่อรันเทสต์ดูก่อน แค่ให้เห็นว่าระบบทำงานอยู่ (ตอนนี้เทสต์ทั้งหมดจะล้มเหลวแน่นอน)

## ข้อกำหนดที่ใช้งานจริง

ในบทนี้เราจะใช้ไลบรารี JavaScript ต่อไปนี้สำหรับการทดสอบ:

- [Mocha](https://mochajs.org/) — เฟรมเวิร์กหลัก ให้ฟังก์ชันทดสอบทั่วไปอย่าง `describe` และ `it` รวมถึงฟังก์ชันสำหรับรันเทสต์
- [Chai](https://www.chaijs.com/) — ไลบรารีที่มี assertion หลากหลายแบบ ตอนนี้เราต้องการแค่ `assert.equal`
- [Sinon](https://sinonjs.org/) — ไลบรารีสำหรับ spy ฟังก์ชัน จำลองการทำงานของฟังก์ชันในตัว และอื่นๆ เราจะใช้ในภายหลัง

ไลบรารีเหล่านี้ใช้ทดสอบได้ทั้งฝั่งเบราว์เซอร์และเซิร์ฟเวอร์ แต่ที่นี่จะพูดถึงกรณีบนเบราว์เซอร์

หน้า HTML แบบเต็มที่รวมเฟรมเวิร์กเหล่านี้พร้อม spec ของ `pow`:

```html src="index.html"
```

หน้านี้แบ่งได้เป็น 5 ส่วน:

1. `<head>` — โหลดไลบรารีจากภายนอกและสไตล์สำหรับเทสต์
2. `<script>` ที่มีฟังก์ชันที่ต้องการทดสอบ — ในที่นี้คือโค้ดของ `pow`
3. เทสต์ — อยู่ในสคริปต์ภายนอก `test.js` ที่มี `describe("pow", ...)` ตามที่เห็นด้านบน
4. `<div id="mocha">` — Mocha จะใช้เพื่อแสดงผลลัพธ์
5. เทสต์จะเริ่มทำงานด้วยคำสั่ง `mocha.run()`

## ปรับปรุงข้อกำหนด

สิ่งที่เราทำไปนั่นถือเป็นการโกงอย่างแน่นอน ฟังก์ชันไม่ได้ทำงานจริง — ลองคำนวณ `pow(3,4)` จะได้ผลลัพธ์ผิด แต่เทสต์กลับผ่าน

...สถานการณ์แบบนี้พบได้บ่อยในทางปฏิบัติ เทสต์ผ่านแต่ฟังก์ชันทำงานผิด แสดงว่า spec ยังไม่ครบ ต้องเพิ่มกรณีการใช้งานเข้าไปอีก

ลองเพิ่มเทสต์อีกหนึ่งอัน เพื่อเช็คว่า `pow(3, 4) = 81`

เราจัดเทสต์ตรงนี้ได้สองแบบ:

1. แบบแรก — เพิ่ม `assert` อีกอันใน `it` เดิม:

   ```js
   describe("pow", function() {

     it("ยกกำลัง n", function() {
       assert.equal(pow(2, 3), 8);
   *!*
       assert.equal(pow(3, 4), 81);
   */!*
     });

   });
   ```

2. แบบที่สอง — แยกเป็นสองเทสต์:

   ```js
   describe("pow", function() {

     it("2 ยกกำลัง 3 เท่ากับ 8", function() {
       assert.equal(pow(2, 3), 8);
     });

     it("3 ยกกำลัง 4 เท่ากับ 81", function() {
       assert.equal(pow(3, 4), 81);
     });

   });
   ```

ความแตกต่างหลักคือ เมื่อ `assert` ฟ้อง error บล็อก `it` จะจบลงทันที ดังนั้นในแบบแรก ถ้า `assert` แรกล้มเหลว เราจะไม่มีทางเห็นผลของ `assert` ตัวที่สองเลย

การแยกเทสต์ออกจากกันช่วยให้เห็นข้อมูลมากขึ้นว่ากำลังเกิดอะไรขึ้น แบบที่สองจึงดีกว่า

นอกจากนั้น ยังมีอีกกฎหนึ่งที่ควรปฏิบัติตาม

**หนึ่งการทดสอบตรวจสอบหนึ่งสิ่ง**

ถ้ามองดูเทสต์แล้วเห็นการตรวจสอบสองอย่างที่ไม่เกี่ยวกัน ควรแยกออกเป็นสองเทสต์ย่อยๆ จะดีกว่า

เรามาใช้แบบที่สองกันต่อเลย

## ปรับปรุงการอิมพลีเมนต์

ลองเขียนอะไรที่เป็นจริงมากขึ้นเพื่อให้เทสต์ผ่านกัน:

```js
function pow(x, n) {
  let result = 1;

  for (let i = 0; i < n; i++) {
    result *= x;
  }

  return result;
}
```

เพื่อให้มั่นใจว่าฟังก์ชันทำงานถูกต้อง ลองทดสอบกับค่าเพิ่มเติมอีก แทนที่จะเขียนบล็อก `it` ด้วยมือ ใช้ลูป `for` ช่วยสร้างได้เลย:

```js
describe("pow", function() {

  function makeTest(x) {
    let expected = x * x * x;
    it(`${x} ยกกำลัง 3 เท่ากับ ${expected}`, function() {
      assert.equal(pow(x, 3), expected);
    });
  }

  for (let x = 1; x <= 5; x++) {
    makeTest(x);
  }

});
```

## การซ้อน describe

เรากำลังจะเพิ่มเทสต์อีกหลายอย่าง แต่ก่อนอื่น สังเกตว่าฟังก์ชันตัวช่วย `makeTest` กับ `for` ควรจัดกลุ่มไว้ด้วยกัน เพราะ `makeTest` ไม่ได้ใช้ที่อื่น — จำเป็นแค่ใน `for` เท่านั้น หน้าที่ร่วมกันคือตรวจว่า `pow` ยกกำลังตามค่าที่กำหนดได้ถูกต้องหรือไม่

การจัดกลุ่มทำได้โดยใช้ `describe` ที่ซ้อนกัน:

```js
describe("pow", function() {

*!*
  describe("ยก x กำลัง 3", function() {
*/!*

    function makeTest(x) {
      let expected = x * x * x;
      it(`${x} ยกกำลัง 3 เท่ากับ ${expected}`, function() {
        assert.equal(pow(x, 3), expected);
      });
    }

    for (let x = 1; x <= 5; x++) {
      makeTest(x);
    }

*!*
  });
*/!*

  // ... เทสต์อื่นๆ จะต่อท้ายที่นี่ โดยเพิ่มทั้ง describe และ it ได้
});
```

`describe` ที่ซ้อนกันจะสร้าง "กลุ่มย่อย" ของเทสต์ขึ้นมา ในผลลัพธ์จะเห็นเป็นหัวข้อย่อยที่เยื้องเข้าไป:

ต่อไปจะเพิ่ม `it` และ `describe` ในระดับบนสุดก็ได้ พร้อมฟังก์ชันตัวช่วยของตัวเอง โดยไม่เห็น `makeTest`

```smart header="`before/after` และ `beforeEach/afterEach`"
ตั้งค่าฟังก์ชัน `before/after` ให้ทำงานก่อน/หลังการรันเทสต์ทั้งหมดได้ รวมถึง `beforeEach/afterEach` ที่จะทำงานก่อน/หลัง `it` *ทุกตัว*

ตัวอย่างเช่น:

```js no-beautify
describe("test", function() {

  before(() => alert("เริ่มการทดสอบ – ก่อนเทสต์ทั้งหมด"));
  after(() => alert("การทดสอบเสร็จสิ้น – หลังเทสต์ทั้งหมด"));

  beforeEach(() => alert("ก่อนเทสต์ – เข้าสู่เทสต์"));
  afterEach(() => alert("หลังเทสต์ – ออกจากเทสต์"));

  it('test 1', () => alert(1));
  it('test 2', () => alert(2));

});
```

ก่อนเทสต์ – เข้าสู่เทสต์ (beforeEach)
2 
หลังเทสต์ – ออกจากเทสต์   (afterEach)
การทดสอบเสร็จสิ้น – หลังเทสต์ทั้งหมด (after)
```

[edit src="beforeafter" title="เปิดตัวอย่างใน sandbox"]

โดยทั่วไป `beforeEach/afterEach` และ `before/after` มักใช้เตรียมสภาพแวดล้อม รีเซ็ตตัวนับ หรือจัดการอะไรบางอย่างระหว่างเทสต์ (หรือระหว่างกลุ่มเทสต์)
````

## ขยายข้อกำหนด

ฟังก์ชันหลักของ `pow` เสร็จแล้ว การพัฒนารอบแรกจบลง เมื่อฉลองดื่มแชมเปญเสร็จ ก็ถึงเวลาพัฒนาต่อ

อย่างที่บอกไป ฟังก์ชัน `pow(x, n)` มีไว้ใช้กับค่า `n` ที่เป็นจำนวนเต็มบวก

เวลาเจอค่าทางคณิตศาสตร์ที่ไม่ถูกต้อง ฟังก์ชันใน JavaScript มักคืนค่า `NaN` — เรามาทำแบบเดียวกันสำหรับค่า `n` ที่ไม่ถูกต้อง

อย่างแรก เพิ่มพฤติกรรมนี้ลงใน spec ก่อน (!):

```js
describe("pow", function() {

  // ...

  it("สำหรับ n ติดลบ ผลลัพธ์คือ NaN", function() {
*!*
    assert.isNaN(pow(2, -1));
*/!*
  });

  it("สำหรับ n ที่ไม่ใช่จำนวนเต็ม ผลลัพธ์คือ NaN", function() {
*!*
    assert.isNaN(pow(2, 1.5));    
*/!*
  });

});
```

เทสต์ใหม่ล้มเหลว เพราะอิมพลีเมนต์ยังไม่รองรับกรณีเหล่านี้ นี่แหละคือวิธี BDD — เขียนเทสต์ที่ล้มเหลวก่อน แล้วค่อยเขียนอิมพลีเมนต์ให้ผ่าน

```smart header="Assertion อื่นๆ"
สังเกตการใช้ assertion `assert.isNaN`: มันตรวจสอบค่า `NaN`

ใน [Chai](https://www.chaijs.com/) ยังมี assertion อื่นๆ อีกด้วย เช่น:

- `assert.equal(value1, value2)` -- ตรวจสอบความเท่ากัน `value1 == value2`
- `assert.strictEqual(value1, value2)` -- ตรวจสอบความเท่ากันแบบเข้มงวด `value1 === value2`
- `assert.notEqual`, `assert.notStrictEqual` -- ตรวจสอบแบบตรงข้ามกับด้านบน
- `assert.isTrue(value)` -- ตรวจสอบว่า `value === true`
- `assert.isFalse(value)` -- ตรวจสอบว่า `value === false`
- ...ดูรายการทั้งหมดได้ใน [เอกสาร](https://www.chaijs.com/api/assert/)
```

ดังนั้นเราควรเพิ่มโค้ดอีกสองสามบรรทัดใน `pow`:

```js
function pow(x, n) {
*!*
  if (n < 0) return NaN;
  if (Math.round(n) != n) return NaN;
*/!*

  let result = 1;

  for (let i = 0; i < n; i++) {
    result *= x;
  }

  return result;
}
```

## สรุป

ใน BDD ข้อกำหนดจะมาก่อน ตามด้วยการอิมพลีเมนต์ ท้ายที่สุดเราจะได้ทั้งข้อกำหนดและโค้ด

ข้อกำหนดสามารถใช้ประโยชน์ได้ 3 ทาง:

1. เป็น **เทสต์** - รับรองว่าโค้ดทำงานถูกต้อง
2. เป็น **เอกสาร** -- ชื่อของ `describe` และ `it` จะบอกว่าฟังก์ชันทำอะไร
3. เป็น **ตัวอย่าง** -- เทสต์จริงๆ แล้วคือตัวอย่างการใช้งานที่แสดงวิธีใช้ฟังก์ชัน

เมื่อมีข้อกำหนด เราสามารถปรับปรุง เปลี่ยนแปลง หรือแม้แต่เขียนฟังก์ชันใหม่ทั้งหมดได้อย่างมั่นใจ และแน่ใจว่ามันยังทำงานได้ถูกต้อง

สิ่งนี้สำคัญมากโดยเฉพาะในโปรเจ็กต์ขนาดใหญ่ที่มีการเรียกใช้ฟังก์ชันในหลายๆ ที่ เมื่อเราแก้ไขฟังก์ชันเหล่านั้น ไม่มีทางเลยที่จะตรวจสอบด้วยมือได้ว่าทุกที่ที่ใช้มันยังทำงานถูกต้องอยู่หรือเปล่า

หากไม่มีเทสต์ ผู้คนมีทางเลือกอยู่สองทาง:

1. ทำการเปลี่ยนแปลงโดยไม่คิดอะไรมาก แล้วผู้ใช้ของเราก็จะเจอบั๊ก เพราะเราอาจพลาดไม่ได้เช็คบางอย่างด้วยตนเอง
2. หรือถ้าการลงโทษข้อผิดพลาดรุนแรง เนื่องจากไม่มีเทสต์ คนก็จะกลัวที่จะแก้ไขฟังก์ชันพวกนั้น ทำให้โค้ดกลายเป็นตัวล้าหลัง ไม่มีใครอยากยุ่งด้วย ไม่เป็นผลดีต่อการพัฒนา

**การทดสอบอัตโนมัติช่วยหลีกเลี่ยงปัญหาเหล่านี้!**

ถ้าโปรเจ็กต์มีเทสต์ที่ครอบคลุม ก็จะไม่มีปัญหาแบบนี้เกิดขึ้น หลังจากมีการเปลี่ยนแปลงใดๆ เราสามารถรันเทสต์และเห็นการตรวจสอบจำนวนมากทำเสร็จภายในไม่กี่วินาที

**อีกทั้ง โค้ดที่ผ่านการทดสอบอย่างดีมักจะมีสถาปัตยกรรมที่ดีกว่าด้วย**

แน่นอนว่าเป็นเพราะโค้ดที่ผ่านการทดสอบอัตโนมัติจะแก้ไขและปรับปรุงได้ง่ายกว่า แต่ก็มีอีกเหตุผลหนึ่ง

การจะเขียนเทสต์ โค้ดจะต้องถูกจัดการให้ฟังก์ชันแต่ละตัวมีหน้าที่ที่ระบุชัดเจน พร้อมอินพุตเอาต์พุตที่กำหนดไว้ดี นั่นหมายถึงการมีสถาปัตยกรรมที่ดีตั้งแต่แรก

ในชีวิตจริง บางครั้งการเขียนข้อกำหนดก่อนเขียนโค้ดจริงก็ไม่ง่ายเสมอไป เพราะอาจยังไม่แน่ชัดว่ามันควรจะทำงานอย่างไร แต่โดยรวมแล้ว การเขียนเทสต์ทำให้การพัฒนาเร็วขึ้นและมั่นคงขึ้น

ในบทเรียนต่อๆ ไป คุณจะได้พบกับหลายๆ แบบฝึกหัดที่มีเทสต์แนบมาด้วย ดังนั้นคุณจะได้เห็นตัวอย่างเชิงปฏิบัติมากขึ้น

การเขียนเทสต์ต้องอาศัยความรู้ JavaScript ที่ดี แต่เรากำลังเพิ่งเริ่มต้นเรียนรู้มัน ดังนั้น เพื่อให้ทุกอย่างเข้าที่เข้าทาง ตอนนี้ยังไม่บังคับให้คุณต้องเขียนเทสต์ แต่ควรอ่านเทสต์ออกบ้างแล้ว ถึงแม้มันจะซับซ้อนกว่าในบทนี้สักหน่อยก็ตาม
