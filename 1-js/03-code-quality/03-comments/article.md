# คอมเมนต์

จากบท <info:structure> เรารู้แล้วว่าคอมเมนต์เขียนได้สองแบบ — แบบบรรทัดเดียวด้วย `//` หรือแบบหลายบรรทัดด้วย `/* ... */`

ปกติแล้วเราใช้คอมเมนต์เพื่ออธิบายว่าโค้ดทำงานอย่างไร และทำไมถึงเขียนแบบนั้น

ฟังดูเหมือนเรื่องง่ายใช่ไหม? แต่มือใหม่มักใช้คอมเมนต์ผิดวิธีอยู่บ่อยๆ

## คอมเมนต์ที่ไม่ดี

มือใหม่มักใช้คอมเมนต์เพื่ออธิบาย "สิ่งที่กำลังเกิดขึ้นในโค้ด" แบบนี้:

```js
// โค้ดนี้จะทำสิ่งนี้ (...) และสิ่งนั้น (...)
// ...และใครจะรู้ว่ามันยังทำอะไรอีก...
very;
complex;
code;
```

แต่ในโค้ดที่ดี คอมเมนต์แบบ "อธิบาย" นี้ควรมีน้อยที่สุด โค้ดควรอ่านเข้าใจได้เองโดยไม่ต้องพึ่งคอมเมนต์

มีกฎที่ดีมากเกี่ยวกับเรื่องนี้ — "ถ้าโค้ดไม่ชัดเจนจนต้องใช้คอมเมนต์ บางทีควรเขียนโค้ดใหม่แทนจะดีกว่า"

### เคล็ดลับ: แยกโค้ดออกมาเป็นฟังก์ชัน

บางครั้งการแยกส่วนหนึ่งของโค้ดออกมาเป็นฟังก์ชันก็ช่วยได้ ลองดูตัวอย่างนี้:

```js
function showPrimes(n) {
  nextPrime:
  for (let i = 2; i < n; i++) {
  
*!*
    // ตรวจสอบว่า i เป็นจำนวนเฉพาะหรือไม่
    for (let j = 2; j < i; j++) {
      if (i % j == 0) continue nextPrime;
    }
*/!*

    alert(i);
  }
}
```

ถ้าแยกออกมาเป็นฟังก์ชัน `isPrime` จะดีกว่า:

```js
function showPrimes(n) {

  for (let i = 2; i < n; i++) {
    *!*if (!isPrime(i)) continue;*/!*

    alert(i);
  }
}

function isPrime(n) {
  for (let i = 2; i < n; i++) {
    if (n % i == 0) return false;
  }

  return true;
}
```

แบบนี้อ่านเข้าใจง่ายขึ้นเลยใช่ไหม? ตัวฟังก์ชันเองก็ทำหน้าที่เป็นคอมเมนต์ไปในตัว โค้ดแบบนี้เรียกว่า *อธิบายตัวเอง (self-descriptive)*

### เคล็ดลับ: สร้างฟังก์ชัน

หากเรามี "แผ่นโค้ด" ยาวๆ แบบนี้:

```js
// ตรงนี้เติมวิสกี้
for(let i = 0; i < 10; i++) {
  let drop = getWhiskey();
  smell(drop);
  add(drop, glass);
}

// ตรงนี้เติมน้ำผลไม้ 
for(let t = 0; t < 3; t++) {
  let tomato = getTomato();
  examine(tomato);
  let juice = press(tomato);
  add(juice, glass);
}

// ...  
```

การ refactor ให้เป็นฟังก์ชันจะดีกว่า:

```js
addWhiskey(glass);
addJuice(glass);

function addWhiskey(container) {
  for(let i = 0; i < 10; i++) {
    let drop = getWhiskey();
    //...
  }
}

function addJuice(container) {
  for(let t = 0; t < 3; t++) {
    let tomato = getTomato();
    //...
  }
}  
```

เช่นเดียวกัน ตัวฟังก์ชันเองบอกอยู่แล้วว่ากำลังทำอะไร ไม่ต้องใช้คอมเมนต์อธิบาย แถมโครงสร้างโค้ดก็ดีขึ้นด้วย — มองเห็นชัดเจนว่าแต่ละฟังก์ชันทำอะไร รับอะไร และคืนค่าอะไร

แน่นอนว่าในความเป็นจริง เราหลีกเลี่ยงคอมเมนต์แบบ "อธิบาย" ไม่ได้ทั้งหมด บางทีมีอัลกอริธึมที่ซับซ้อน หรือมี "กลเม็ด" เพื่อเพิ่มประสิทธิภาพโค้ด แต่โดยทั่วไปแล้ว ควรพยายามเขียนโค้ดให้เรียบง่ายและอธิบายตัวเองได้มากที่สุด

## คอมเมนต์ที่ดี

ดังนั้น คอมเมนต์ที่เป็นคำอธิบายมักจะไม่ดี แล้วคอมเมนต์แบบไหนถึงจะดีล่ะ?

อธิบายสถาปัตยกรรม
: ให้ภาพรวมของส่วนประกอบต่างๆ ว่าทำงานร่วมกันอย่างไร ลำดับการทำงานในแต่ละสถานการณ์เป็นอย่างไร — พูดง่ายๆ คือมองโค้ดจากมุมสูง มีภาษาอย่าง [UML](http://wikipedia.org/wiki/Unified_Modeling_Language) ที่ใช้สร้างแผนภาพสถาปัตยกรรมเพื่ออธิบายโค้ด ซึ่งคุ้มค่าแก่การศึกษามาก

บันทึกพารามิเตอร์และการใช้งานฟังก์ชัน
: มีไวยากรณ์พิเศษอย่าง [JSDoc](http://en.wikipedia.org/wiki/JSDoc) สำหรับบันทึกข้อมูลฟังก์ชัน — วิธีใช้งาน พารามิเตอร์ และค่าที่คืนกลับมา

เช่น:
```js
/**
 * คืนค่า x ยกกำลัง n
 *
 * @param {number} x เลขที่ต้องการยกกำลัง
 * @param {number} n เลขชี้กำลัง ต้องเป็นจำนวนนับ
 * @return {number} x ยกกำลัง n
 */
function pow(x, n) {
  ...
}
```

คอมเมนต์แบบนี้ช่วยให้เข้าใจจุดประสงค์ของฟังก์ชันและใช้งานได้ถูกต้อง โดยไม่ต้องไปอ่านโค้ดข้างใน

นอกจากนี้ เครื่องมือแก้ไขโค้ดหลายตัวอย่าง [WebStorm](https://www.jetbrains.com/webstorm/) ก็อ่านคอมเมนต์เหล่านี้ได้ แล้วนำไปช่วยเติมโค้ดอัตโนมัติและตรวจสอบโค้ดให้ด้วย

ยังมีเครื่องมืออย่าง [JSDoc 3](https://github.com/jsdoc/jsdoc) ที่สร้างเอกสาร HTML จากคอมเมนต์ได้โดยตรง อ่านรายละเอียดเพิ่มเติมเกี่ยวกับ JSDoc ได้ที่ <https://jsdoc.app>

ทำไมปัญหาถึงแก้ด้วยวิธีนี้?
: สิ่งที่เขียนไว้นั้นสำคัญ แต่สิ่งที่ *ไม่ได้* เขียนอาจสำคัญยิ่งกว่า — ทำไมถึงเลือกแก้ปัญหาด้วยวิธีนี้? โค้ดไม่ได้บอกคำตอบ

  ถ้ามีหลายวิธีแก้ปัญหา ทำไมถึงเลือกวิธีนี้? โดยเฉพาะเมื่อไม่ใช่วิธีที่ตรงไปตรงมาที่สุด

  ถ้าไม่มีคอมเมนต์อธิบาย สถานการณ์แบบนี้อาจเกิดขึ้นได้:
  1. เรา (หรือเพื่อนร่วมงาน) เปิดโค้ดที่เขียนไว้เมื่อสักพัก แล้วเห็นว่า "ไม่เหมาะสมเท่าไร"
  2. คิดว่า "ตอนนั้นโง่จัง ตอนนี้ฉลาดขึ้นแล้ว" แล้วก็เขียนใหม่ด้วยวิธีที่ "ตรงไปตรงมาและถูกต้องกว่า"
  3. ...ความอยากเขียนใหม่เป็นเรื่องดี แต่ระหว่างทำก็พบว่าวิธีที่ "ตรงไปตรงมา" จริงๆ แล้วไม่ครบถ้วน พอจำได้ลางๆ ว่าทำไมถึงเป็นแบบเดิม เพราะเคยลองมาแล้วนานมาแล้ว สุดท้ายก็เปลี่ยนกลับไปใช้วิธีเดิม — แต่เสียเวลาไปเปล่าๆ

  คอมเมนต์ที่อธิบายเหตุผลของวิธีแก้ปัญหาจึงสำคัญมาก ช่วยให้พัฒนาต่อไปในทิศทางที่ถูกต้อง

โค้ดมีจุดละเอียดอ่อนหรือเทคนิคพิเศษซ่อนอยู่ไหม? ใช้ตรงไหนบ้าง?
: ถ้าโค้ดมีอะไรที่ละเอียดอ่อนหรือขัดกับสัญชาตญาณ ก็ควรเขียนคอมเมนต์ไว้ให้ชัดเจน

## สรุป

สัญญาณสำคัญของนักพัฒนาที่ดีคือคอมเมนต์ — ทั้งการมีและการไม่มี

คอมเมนต์ที่ดีช่วยให้ดูแลโค้ดได้ดี กลับมาอ่านทีหลังก็ยังเข้าใจ และใช้งานได้อย่างถูกต้อง

**ควรเขียนคอมเมนต์เรื่อง:**

- สถาปัตยกรรมโดยรวม มุมมองภาพกว้าง
- วิธีใช้งานฟังก์ชัน
- วิธีแก้ปัญหาที่สำคัญ โดยเฉพาะเมื่อเหตุผลไม่ได้ชัดเจนในทันที

**หลีกเลี่ยงคอมเมนต์ที่:**

- บอกว่า "โค้ดทำงานอย่างไร" หรือ "ทำอะไร"
- ใส่คอมเมนต์ก็ต่อเมื่อไม่สามารถทำให้โค้ดเรียบง่ายและอธิบายตัวเองได้จริงๆ เท่านั้น

คอมเมนต์ยังใช้กับเครื่องมือสร้างเอกสารอัตโนมัติอย่าง JSDoc3 ได้ด้วย — เครื่องมือจะอ่านคอมเมนต์แล้วสร้างเอกสาร HTML (หรือรูปแบบอื่น) ให้เลย
