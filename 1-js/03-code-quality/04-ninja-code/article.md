# โค้ดนินจา

```quote author="ขงจื๊อ (วจนะ)"
เรียนไร้คิดเหนื่อยเปล่าดุจดั่งฝัน คิดไร้เรียนอันตรายใช่น้อยเลย
```

โปรแกรมเมอร์นินจาในอดีตใช้กลเม็ดเหล่านี้เพื่อลับสมองของผู้ดูแลโค้ด

ผู้เชี่ยวชาญการรีวิวโค้ดมองหากลเม็ดเหล่านี้ในการทดสอบงาน

บางครั้งนักพัฒนามือใหม่ใช้กลเม็ดเหล่านี้ได้ดีกว่าโปรแกรมเมอร์นินจาเสียอีก

ลองอ่านดูให้ดีแล้วถามตัวเองว่าเราเป็นใคร — นินจา, มือใหม่, หรืออาจเป็นผู้รีวิวโค้ด?

```warn header="พบการเสียดสี"
มากมายมุ่งวิถีนินจา น้อยนักล่วงพ้นถึงจุดหมาย
```

## ความกระชับคือหัวใจของปัญญา

เขียนโค้ดให้สั้นที่สุดเท่าที่จะทำได้ โชว์ความฉลาดให้โลกเห็น

ปล่อยให้ฟีเจอร์ลึกลับของภาษาเป็นตัวนำทาง

เช่น ลองดู ternary operator `'?'` นี้:

```js
// มาจากไลบรารี JavaScript ที่มีชื่อเสียง
i = i ? i < 0 ? Math.max(0, len + i) : i : 0;
```

เจ๋งใช่ไหม? ถ้าเขียนแบบนี้ นักพัฒนาที่เจอบรรทัดนี้แล้วพยายามไล่ดูว่า `i` มีค่าเท่าไหร่ คงจะสนุกกันน่าดู จากนั้นก็ต้องมาถามเราเพื่อขอคำตอบ

บอกไปเลยว่าสั้นกว่าดีกว่าเสมอ แล้วชี้ทางพวกเขาสู่วิถีแห่งนินจา

## ตัวแปรตัวอักษรเดียว

```quote author="เล่าจื๊อ (เต๋าเต๋อจิง)"
เต๋าซ่อนในความเงียบไร้คำ เริ่มต้นและจบลงอย่างสมบูรณ์
```

อีกวิธีหนึ่งที่จะเขียนโค้ดให้สั้นลงคือการใช้ชื่อตัวแปรเพียงตัวอักษรเดียวทุกที่ เช่น `a`, `b` หรือ `c`

ตัวแปรที่สั้นจะหายไปในโค้ดเหมือนนินจาหายเข้าป่า ไม่มีใครหาเจอด้วยฟีเจอร์ "ค้นหา" ของ editor ถึงจะเจอก็ "ถอดรหัส" ไม่ออกว่า `a` หรือ `b` หมายถึงอะไร

...แต่มีข้อยกเว้น นินจาตัวจริงจะไม่มีวันใช้ `i` เป็นตัวนับในลูป `"for"` ที่ไหนก็ใช้ได้ แต่ไม่ใช่ตรงนี้ ลองมองหารอบๆ ดูสิ มีตัวอักษรแปลกๆ อีกเพียบ เช่น `x` หรือ `y`

ใช้ตัวแปรแปลกตาเป็นตัวนับลูปนี่เจ๋งสุดๆ โดยเฉพาะถ้าภายในลูปยาว 1-2 หน้า (ยิ่งยาวยิ่งดี) เพราะถ้ามีใครมาไล่ดูโค้ดข้างในลูป จะเดาไม่ออกเลยว่าตัวแปรชื่อ `x` คือตัวนับลูปนั่นเอง


## ใช้คำย่อ

หากกฎของทีมห้ามใช้ชื่อที่เป็นตัวอักษรเดียวหรือคลุมเครือ — ก็ย่อซะ สร้างคำย่อขึ้นมา

แบบนี้:

- `list` -> `lst`
- `userAgent` -> `ua`
- `browser` -> `brsr` 
- ...ฯลฯ

มีเพียงคนที่สัญชาตญาณดีเยี่ยมเท่านั้นถึงจะเข้าใจชื่อแบบนี้ได้ พยายามย่อทุกอย่าง มีแต่คนที่คู่ควรเท่านั้นที่จะดูแลโค้ดของเราต่อได้

## โบยบินสูงส่ง ดั่งนามธรรม

```quote author="เล่าจื๊อ (เต๋าเต๋อจิง)"
สี่เหลี่ยมใหญ่ไร้มุมแหลมคม
ภาชนะยิ่งใหญ่เสร็จสิ้นท้ายสุด
เสียงอันไพเราะคือความเบาบาง
ภาพอันยิ่งใหญ่ไร้รูปทรง
```

ตอนตั้งชื่อ พยายามเลือกคำที่เป็นนามธรรมที่สุด เช่น `obj`, `data`, `value`, `item`, `elem` เป็นต้น

- **`data` คือชื่อตัวแปรในอุดมคติ** ใช้ทุกที่ที่ทำได้เลย เพราะตัวแปรทุกตัวก็เก็บ *ข้อมูล* อยู่แล้วจริงไหม?

    ...แต่ถ้า `data` ถูกจองไปแล้วล่ะ? ลองใช้ `value` แทน เพราะก็เป็นคำสากลเหมือนกัน ท้ายที่สุดตัวแปรก็ต้องมี *ค่า* อยู่ดี

- **ตั้งชื่อตัวแปรตามชนิดข้อมูล: `str`, `num`...**

    ลองใช้ดู มือใหม่อาจสงสัยว่าชื่อแบบนั้นมีประโยชน์จริงหรือสำหรับนินจา? แน่นอน มีประโยชน์จริงๆ!

    ชื่อตัวแปรก็ยังพอบอกอะไรได้บ้าง — บอกว่าข้างในเก็บอะไร: สตริง ตัวเลข หรืออย่างอื่น แต่พอคนนอกมาอ่านโค้ด จะพบว่าชื่อแบบนี้ไม่ได้ให้ข้อมูลอะไรเลย! แล้วก็จะแก้โค้ดที่เราคิดมาอย่างดีไม่ได้สักที

    ชนิดข้อมูลหาได้ง่ายจากการดีบัก แต่ความหมายของตัวแปรล่ะ? เก็บสตริง/ตัวเลขอะไรกันแน่?

    ไม่มีทางรู้ได้เลย นอกจากจะทำสมาธิให้ดี!

- **...แล้วถ้าชื่อแบบนั้นไม่เหลือแล้วล่ะ?** ก็แค่เติมตัวเลขต่อท้าย: `data1, item2, elem5`...

## ทดสอบความใส่ใจ

มีแค่โปรแกรมเมอร์ที่ใส่ใจจริงๆ เท่านั้นถึงจะอ่านโค้ดของเราออก แต่จะทดสอบได้ยังไงล่ะ?

**วิธีหนึ่งคือใช้ชื่อตัวแปรที่คล้ายกัน เช่น `date` กับ `data`**

ใช้สลับกันไปมาตามใจชอบเลย

แค่นี้ก็อ่านโค้ดผ่านๆ ไม่ได้แล้ว แล้วถ้าดันมีพิมพ์ผิดอีก... อืม... คงได้ติดอยู่นาน ถึงเวลาพักไปชงชากันแล้ว

## คำพ้องความหมายสุดเจ๋ง

```quote author="เล่าจื๊อ (เต๋าเต๋อจิง)"  
เต๋าที่กล่าวได้มิใช่เต๋านิรันดร์ นามที่เอ่ยได้มิใช่นามอมตะ
```

การใช้ชื่อ *คล้ายๆ กัน* กับสิ่งที่ *เหมือนๆ กัน* ทำให้ชีวิตสนุกขึ้นและยังโชว์ความคิดสร้างสรรค์ให้คนอื่นเห็นด้วย

เช่น คำนำหน้าฟังก์ชัน ถ้าฟังก์ชันแสดงข้อความบนจอ ให้ขึ้นต้นด้วย `display...` อย่าง `displayMessage` แล้วถ้ามีอีกฟังก์ชันที่แสดงอย่างอื่นบนจอ เช่นชื่อผู้ใช้ ก็ขึ้นต้นด้วย `show...` แทน (อย่าง `showName`)

ทำให้ดูเหมือนมีความต่างอย่างละเอียดอ่อน ทั้งที่จริงๆ แล้วไม่ได้ต่างกันเลย

นัดแนะกับเพื่อนนินจาในทีมเลย — ถ้าจอห์นใช้ `display...` ในโค้ดของเขา ปีเตอร์ก็ใช้ `render...` ส่วนแอนก็ `paint...` ดูสิว่าโค้ดจะน่าสนใจและหลากหลายขึ้นแค่ไหน

...แล้วนี่คือไม้เด็ด!

สำหรับสองฟังก์ชันที่แตกต่างกันชัดเจน ให้ใช้คำนำหน้าเดียวกัน!

เช่น ฟังก์ชัน `printPage(page)` จะส่งไปเครื่องพิมพ์ ส่วน `printText(text)` จะแสดงข้อความบนจอ ให้คนอ่านที่ไม่คุ้นเคยได้คิดหนักกับ `printMessage` ว่า "ส่งข้อความไปไหนนะ เครื่องพิมพ์หรือบนจอ?" จะเจ๋งขึ้นไปอีกถ้า `printMessage(message)` ดันไปเปิดหน้าต่างใหม่ขึ้นมา!

## ใช้ชื่อซ้ำ

```quote author="เล่าจื๊อ (เต๋าเต๋อจิง)"
เมื่อแบ่งสรรพสิ่งออกเป็นส่วนย่อย
แต่ละส่วนล้วนต้องการชื่อเรียกขาน
นามทั้งหลายมีอยู่มากมายแล้ว
รู้จักพอคือคุณูปการอันล้ำค่า
```

อย่าเพิ่มตัวแปรใหม่ถ้าไม่จำเป็นจริงๆ

แทนที่จะทำแบบนั้น เอาชื่อที่มีอยู่แล้วมาใช้ซ้ำ แค่ยัดค่าใหม่ลงไปทับค่าเดิม

ภายในฟังก์ชัน พยายามใช้เฉพาะตัวแปรที่ส่งเข้ามาเป็นพารามิเตอร์เท่านั้น

วิธีนี้จะทำให้แทบเป็นไปไม่ได้เลยที่จะรู้ว่า *ตอนนี้* ตัวแปรเก็บอะไรอยู่ และมาจากไหน จุดประสงค์คือฝึกสัญชาตญาณและความจำของคนอ่านโค้ด ใครที่สัญชาตญาณไม่ดีก็ต้องไล่วิเคราะห์โค้ดทีละบรรทัด ติดตามการเปลี่ยนแปลงข้ามทุกสาขาของโค้ดไป

**รูปแบบขั้นสูงของวิธีนี้คือการแอบ (!) เปลี่ยนค่าเป็นอย่างอื่นที่คล้ายๆ กันระหว่างวนลูปหรือทำงานในฟังก์ชัน**

ตัวอย่างเช่น:

```js
function ninjaFunction(elem) {
  // 20 บรรทัดโค้ดที่ทำงานกับ elem 

  elem = clone(elem);

  // อีก 20 บรรทัด ตอนนี้ทำงานกับ clone ของ elem!
}
```

โปรแกรมเมอร์ที่ต้องการทำงานกับ `elem` ในครึ่งหลังของฟังก์ชันจะงงแน่ๆ... ต้องดีบักไล่ดูโค้ดถึงจะรู้ว่าตัวเองกำลังทำงานกับตัวโคลนอยู่!

เจอแบบนี้ในโค้ดบ่อยมาก ได้ผลดีถึงตายแม้กับนินจาจอมเก๋า

## ใช้ underscore ให้สนุก

ใส่ underscore `_` และ `__` ข้างหน้าชื่อตัวแปร เช่น `_name` หรือ `__value` ถ้ามีแค่เราที่รู้ความหมายก็ยิ่งดี หรือดีกว่านั้น — ใส่ไปเพื่อความสนุกเฉยๆ โดยไม่มีความหมายอะไรเลย หรือจะให้ความหมายต่างกันในแต่ละที่ก็ได้

แค่นี้ก็ยิงปืนนัดเดียวได้นกสองตัว หนึ่ง โค้ดจะยาวขึ้นและอ่านยากขึ้น สอง เพื่อนนักพัฒนาอาจเสียเวลาไปเยอะกับการพยายามหาว่า underscore หมายถึงอะไร

นินจาเก๋าจะวาง underscore ไว้ตรงนึง แล้วไม่ใส่อีกตรงนึง แค่นี้ก็ยิ่งทำให้โค้ดเปราะบางและเพิ่มโอกาสเกิดบั๊กในอนาคตแล้ว

## แสดงความรัก

ให้ทุกคนได้เห็นความยิ่งใหญ่ของ entity ต่างๆ ที่เราสร้าง! ชื่ออย่าง `superElement`, `megaFrame` และ `niceItem` จะทำให้ผู้อ่านตาสว่างแน่นอน

ก็จริงอยู่ว่ามีอะไรบางอย่างเขียนไว้: `super..`, `mega..`, `nice..` แต่อีกแง่หนึ่ง ก็ไม่ได้ให้รายละเอียดอะไรเลย ผู้อ่านอาจต้องนั่งค้นหาความหมายที่ซ่อนอยู่ นั่งสมาธิสักชั่วโมงสองชั่วโมงในเวลางานที่ได้ค่าจ้าง

## ใช้ชื่อซ้ำกับตัวแปรด้านนอก

```quote author="กวนอินจื้อ"
ท่ามกลางแสงสว่าง มองไม่เห็นสิ่งใดในความมืด<br>
ในความมืด กลับมองเห็นทุกสิ่งในแสงสว่าง
```

ใช้ชื่อเดียวกันสำหรับตัวแปรทั้งในและนอกฟังก์ชัน แค่นี้แหละ ไม่ต้องพยายามคิดชื่อใหม่

```js
let *!*user*/!* = authenticateUser();

function render() {
  let *!*user*/!* = anotherValue();
  ...
  ...หลายบรรทัด...
  ...
  ... // <-- โปรแกรมเมอร์อยากทำงานกับ user ตรงนี้และ... 
  ...
}
```

โปรแกรมเมอร์ที่กระโดดเข้ามาใน `render` อาจไม่ทันสังเกตว่ามี `user` ตัวใหม่ในฟังก์ชันกำลังบดบัง `user` ตัวนอกอยู่

แล้วก็จะพยายามทำงานกับ `user` โดยคิดว่าเป็นตัวแปรจากภายนอก ผลลัพธ์จาก `authenticateUser()`... กับดักถูกวางแล้ว! สวัสดี debugger...

## ผลข้างเคียงได้ทุกที่!

มีฟังก์ชันบางตัวที่ดูเหมือนไม่ได้เปลี่ยนแปลงอะไร เช่น `isReady()`, `checkPermission()`, `findTags()`... ปกติจะคาดหวังว่าฟังก์ชันพวกนี้แค่คำนวณ ค้นหา และคืนข้อมูล โดยไม่แตะต้องสิ่งใดภายนอก — พูดอีกอย่างก็คือ ไม่มี "ผลข้างเคียง"

**กลเม็ดสุดเจ๋งคือแอบใส่การกระทำที่ "มีประโยชน์" เข้าไป นอกเหนือจากงานหลัก**

สีหน้างงงวยของเพื่อนร่วมงาน เมื่อเห็นฟังก์ชันชื่อ `is..`, `check..` หรือ `find...` ที่ดันไปเปลี่ยนแปลงอะไรบางอย่าง จะช่วยขยายขอบเขตแห่งปัญญาได้แน่นอน

**อีกวิธีที่จะสร้างความประหลาดใจคือคืนค่าผลลัพธ์แบบไม่ปกติ**

โชว์ความคิดริเริ่มเลย! ให้ `checkPermission` คืนค่าไม่ใช่แค่ `true/false` แต่เป็นออบเจ็กต์ที่ซับซ้อนพร้อมผลการตรวจสอบ

นักพัฒนาที่พยายามเขียน `if (checkPermission(..))` จะสงสัยว่าทำไมถึงใช้ไม่ได้ บอกไปเลยว่า "อ่าน docs สิ!" แล้วก็ส่งลิงก์บทความนี้ให้

มหาเต๋าไหลทั่วทุกหนแห่ง

เช่น ฟังก์ชัน `validateEmail(email)` นอกจากตรวจสอบความถูกต้องของอีเมลแล้ว ก็อาจแสดง error และขอให้กรอกอีเมลใหม่ด้วย

สิ่งที่ฟังก์ชันทำเพิ่มเติมไม่ควรเดาได้จากชื่อ นินจาตัวจริงจะทำให้มองไม่ออกจากโค้ดเช่นกัน

**การยัดหลายอย่างไว้ในฟังก์ชันเดียวจะปกป้องโค้ดของเราจากการถูกนำไปใช้ซ้ำ**

สมมติว่ามีนักพัฒนาอีกคนต้องการแค่เช็คอีเมลเฉยๆ โดยไม่ต้องแสดงข้อความ ฟังก์ชัน `validateEmail(email)` ของเราที่ทำงานทั้งสองอย่างจะไม่เหมาะกับเขา แบบนี้เขาก็จะไม่มารบกวนสมาธิด้วยการมาถามนู่นนี่

## สรุป

"เคล็ดลับ" ข้างต้นทั้งหมดมาจากโค้ดของจริง... บางครั้งเขียนโดยนักพัฒนาผู้มากประสบการณ์ บางทีอาจจะมากกว่าเราด้วยซ้ำ ;)

- ทำตามบางข้อ โค้ดของเราก็จะเต็มไปด้วยเรื่องน่าประหลาดใจ
- ทำตามหลายๆ ข้อ โค้ดจะกลายเป็นของเราโดยแท้จริง ไม่มีใครอยากแตะต้อง
- ทำตามทุกข้อ โค้ดของเราจะกลายเป็นบทเรียนอันล้ำค่าสำหรับนักพัฒนาหน้าใหม่ผู้แสวงหาการรู้แจ้ง
